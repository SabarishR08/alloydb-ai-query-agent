AlloyDB AI Query Agent - Implementation Progress Report
Date: 2026-03-25
Repository: alloydb-ai-query-agent
Branch: main

============================================================
1) PROJECT OBJECTIVE
============================================================
Build a production-style AI query system where users can ask natural language questions,
translate them into SQL, execute against a PostgreSQL-compatible database, and return
structured JSON results through a deployable API and frontend.

Core target flow:
User -> Frontend -> FastAPI -> SQL generation (Gemini/fallback) -> Cloud SQL -> Results

============================================================
2) WHAT HAS BEEN IMPLEMENTED
============================================================
A) Repository and baseline setup
- Repository initialized and pulled into workspace.
- Copilot-generated starter backend integrated into main branch.
- Python dependencies and Dockerfile included.

B) Backend API (FastAPI)
- Main API service implemented with:
  - POST /query endpoint for natural-language queries
  - GET /health endpoint for health checks
- Input/output modeling done via Pydantic.
- Structured exception handling added for SQL generation and DB execution paths.

C) Database layer
- Database connection module built with psycopg2.
- Read-only query enforcement (SELECT-only execution) in DB execution path.
- Cloud SQL socket support implemented via CLOUD_SQL_CONNECTION_NAME.
- Host/port mode retained for local/direct DB connectivity.

D) SQL generation engine
- Gemini integration implemented (Vertex AI SDK path).
- Prompt/schema context tuned for ai_tools dataset.
- Safety guard retained to reject non-SELECT behavior.
- Resilient local fallback SQL generator added for high availability when Gemini
  model access is unavailable in the active GCP project.
- Runtime flag introduced: USE_GEMINI (true/false) to switch behavior.

E) Dataset and schema
- Custom dataset authored in data/seed.sql.
- ai_tools schema includes:
  - id, name, category, description, popularity_score
- Seed data includes AI tooling categories (Backend, DevOps, Vector DB, LLM API,
  LLM Framework, ML Platform).

F) Frontend
- Responsive frontend implemented at frontend/index.html.
- Features:
  - Query input
  - Search action
  - Generated SQL display
  - Result table rendering
  - Loading and error states
  - Client-side sort by popularity_score desc
- FastAPI serves frontend at root (/).
- Static frontend path mounted for Cloud Run image.

G) Cloud deployment and operations
- Cloud Run service deployed and serving traffic.
- Cloud SQL PostgreSQL instance confirmed and ai_tools_db present.
- Seed SQL imported through Cloud SQL import workflow.
- Cloud Run wired with:
  - Cloud SQL instance attachment
  - Environment variables for DB + runtime controls
- Runtime issue handling completed:
  - Gemini model NotFound in project resolved operationally by enabling deterministic
    fallback SQL path so endpoint continues to function.

H) Documentation and submission assets
- README expanded with setup, deploy, architecture, and validation guidance.
- Submission-oriented sections added.
- Demo script added: scripts/demo.ps1.

============================================================
3) MAJOR ISSUES ENCOUNTERED AND RESOLVED
============================================================
1. Gemini model access errors in active project
- Symptom: "Failed to generate SQL from Gemini" and 404 NotFound for publisher model.
- Resolution: Added robust fallback SQL generator and USE_GEMINI runtime flag.
- Outcome: Query endpoint remains functional even when Vertex model access is restricted.

2. Cloud SQL connectivity and import constraints
- Symptom: direct psql timeout/import permission errors.
- Resolution:
  - Used Cloud Storage staging for seed.sql
  - Granted proper storage access for Cloud SQL import identity
  - Imported dataset server-side into ai_tools_db
- Outcome: Database initialized with live data.

3. Cloud Run env var formatting issue
- Symptom: env vars collapsed into one value; app attempted localhost DB.
- Resolution: Redeployed using env-vars-file format and verified container env.
- Outcome: Correct runtime configuration applied.

4. DB auth mismatch
- Symptom: password authentication failure for postgres.
- Resolution: synchronized Cloud SQL user password with deployed runtime settings.
- Outcome: Successful DB access from deployed service.
